# ISSUE-021：软著设计说明书与用户操作手册的图文覆盖和说明深度不足

**状态**：代码修正完成，待用户视觉验收

**优先级**：P1

**发现日期**：2026-06-23

## 用户反馈

- 生成的设计说明书和用户操作手册偏简单，缺少图文并茂的功能说明。
- 自动截图过少，菜单和功能没有逐项覆盖。
- 功能简介、操作描述过于笼统。
- 可参考用户提供的页面范例和公开模板的结构，但不得复制第三方页面、文案或版式成品。

## 已复现的现状

1. `capture_screenshots()` 固定生成登录页、首页及每个模块的一张列表页截图；只在第一个模块额外生成一张“新增”弹窗截图。
2. `generate_documents()` 仅在用户手册的模块章节插入一张与模块名匹配的截图；设计说明书不插入截图或结构图。
3. 当前每个模块在设计说明书中的内容只有模块描述、页面名和字段名；用户手册只有一句“从左侧菜单进入并查看/维护字段”。
4. `build_compliance_report()` 只校验“每个模块至少有一张截图”和“模块名称出现在两份文档”，未校验页面、业务动作、截图插入数量、操作步骤或图文对应关系，因此该质量缺口不会阻断材料通过。

## 初步根因

- 截图规划粒度停留在“模块”，未以 `planning.json` 的 `modules[].pages` 与 `api_list` 的业务动作为覆盖单元。
- 文档生成器没有“功能单元”模型，无法把功能概述、角色、前置条件、页面字段、操作步骤、结果和截图建立一一对应关系。
- 合规检查定位为完整性检查，缺少材料质量门槛。

## 参考方向（仅借鉴信息结构）

用户提供的范例和公开模板调研体现了同一套有效结构：每个功能章节都应有独立截图、功能简介、编号化操作步骤和流程表达。拟采用以下原创模板，而非复用任何外部模板的封面、页面、文字或截图：

| 材料 | 固定结构 |
| --- | --- |
| 设计说明书 | 项目概述 → 系统架构图 → 功能总览图 → 按模块/页面展开的功能设计 → 数据设计 → 接口设计 → 截图索引 |
| 用户操作手册 | 使用对象与环境 → 登录/首页 → 按菜单顺序的功能单元 → 每个单元的用途、前置条件、编号步骤、结果说明、界面截图与流程图 → 常见提示 |

版式仅采用统一的页眉页脚、自动页码、章节标题、表格、图题和编号；视觉重心放在真实生成项目截图及由规划数据生成的结构图/流程图上。

## 2026-06-24 调研结论与混合生成约束

- 用户提供的本地样例均采用“目录、软件概述、目标/用户、特色、流程、技术特点、运行环境、按功能截图与操作说明”的结构；代码鉴别材料为连续代码页，并保持软件名称、版本和页码一致。
- 用户提供的知乎教程将文档鉴别材料定义为能描述程序内容、组成、设计、功能规格和使用方法的文字资料与图表；重点要求操作手册覆盖登录页、完整主要功能、截图和文字步骤，设计说明覆盖架构、流程、逻辑、接口和模块设计。该教程仅作公开经验参考，最终以中国版权保护中心最新要求为准。
- 实现采用“固定版式与事实证据 + 可选 LLM 文案”的混合模式：LLM 只接收模块、页面、字段、实际接口和截图清单，输出严格 JSON；不得新增事实。结构、图题、截图、申请信息表和 PDF/DOCX 版式均由固定生成器控制。
- 仅显式配置 `AI_DOCUMENT_MODEL` 才会调用文案模型；请求、解析或事实校验失败自动回退固定文案，并写入 `document_narratives.json`。

## 拟议实施范围

### 1. 截图覆盖模型

- 保留登录页与 Dashboard 截图。
- 对每个已确认模块分别截取列表/查询入口、创建或编辑入口、详情或关键业务操作入口；对 `api_list` 中生成的业务动作，至少按模块形成可见的动作覆盖截图。
- 截图名称、元数据和文档引用均使用稳定的“模块 + 页面/动作”标识，避免依赖中文文件名模糊匹配。
- 当规划的 `pages` 与实际固定生成器无法形成独立路由时，应在 ISSUE 实施中明确映射为“模块功能单元”，不得伪造不存在的页面截图。

### 2. 设计说明书增强

- 新增系统架构图、功能模块总览图、模块处理流程图，以及与真实页面对应的截图和图题。
- 每个功能单元至少包含：功能目标、适用角色、输入/输出、业务规则、页面字段、涉及数据表/API、处理流程、界面说明。
- 使用 `planning.json`、代码统计、实际截图和生成接口作为唯一输入；不调用与规划无关的行业模板填充内容。

### 3. 用户操作手册增强

- 每个菜单功能单元至少包含：功能简介、使用前提、操作入口、3–6 个可执行步骤、成功结果/异常提示和对应截图。
- 对新增、查询、详情、状态变更等已生成的核心动作分别说明；没有生成的动作不写成已支持能力。
- 插入流程图或编号流程表，并为每张图片添加“图 x-x”标题和功能对应关系。

### 4. 材料质量校验与测试

- 扩展合规报告：核对全部模块、功能页面/动作的截图覆盖；两份文档的章节、图片与图题数量；每个功能单元的最少步骤数。
- 为截图计划、DOCX 图文结构和质量校验新增自动化测试；测试只使用临时任务目录。
- 使用一个临时生成项目执行截图、DOCX 检查、前端构建和 JDK 17 Maven 测试，确认没有破坏 Demo 审查门禁和材料重试逻辑。

## 非目标

- 不修改既有历史任务的 `outputs/` 内容。
- 不抄袭用户提供截图、第三方模板、第三方文案或封面设计。
- 不扩大到 Planner 行业知识库、项目业务代码或在线 Demo 审查流程的改造。
- 不以虚构截图、虚构操作步骤或未生成页面来凑页数。

## 验收标准

1. 每个规划模块都能在文档中找到至少一组“真实截图 + 功能说明 + 编号操作步骤”；登录和 Dashboard 也有独立图文说明。
2. 设计说明书包含至少系统架构、功能总览、模块功能设计、数据/API 设计和截图索引；模块章节含真实截图或从规划生成的结构图。
3. 用户操作手册覆盖所有可见菜单模块及其已生成的核心操作，不再只有单句描述。
4. 合规报告能够识别缺失的模块截图、功能单元章节或低于最少步骤数的操作说明。
5. 临时生成项目通过前端构建、JDK 17 Maven 测试；材料阶段仍只能在 Demo 审查通过后运行，失败后仍可从 `screenshot` 阶段受控恢复。

## 2026-06-24 产物复核：版式与材料真实性缺陷

对成功任务 `outputs/20260624140839-03bb66f7` 的 DOCX 结构和用户截图复核后，确认此前的图文覆盖实现没有达到最终材料质量要求。本节作为 ISSUE-021 的第二阶段，覆盖以下缺陷。

1. **页码没有渲染。** 四份材料的页眉第二段均只有“第 ”；当前生成的 `w:fldSimple` PAGE 域未被用户所用办公软件解析，导致页面标识不完整。
2. **源代码材料泄露 AI 增强痕迹。** 生成器将 `frontend/src/style.css` 原样纳入代码材料，包含 `AI UI Enhancer` 注释；增强样式又以压缩长行形式写入，既暴露实现来源，也破坏代码页可读性。
3. **代码页版心与分页不稳定。** 当前用普通段落逐行排版并按固定行数插入分页，遇到超长 CSS/XML 行会发生视觉换行，导致一页顶部只有少量内容、下方大面积留白，不能保证连续、清晰的代码页。
4. **用户操作手册仍然过短且版式不符合参考结构。** 每个功能仍以简短模板文本和四列表格替代完整操作说明。样例所要求的“功能简介 → 操作说明 → 真实截图/图题”主线没有落实；表格不能代替逐步、可执行的文字说明。

### 第二阶段修复范围（待用户确认后实施）

- 建立并强制应用材料样式规范，而不是由各段落临时设置字体：封面、一级/二级/三级标题、正文、代码、表格、图题、页眉页脚分别定义中文字体、西文字体、字号、加粗、缩进、行距、段前段后和对齐方式。源代码材料使用等宽代码字体；设计说明书与用户手册使用统一的中文文档字体，避免同页混用或默认字体回退。
- 将样式规范作为生成器唯一入口：标题、图片说明、表格单元格和页码不得绕过样式；对 DOCX 的样式名、字体和字号进行结构化检查，防止后续功能插入时重新出现默认字体、异常字号或不一致的行距。
- 用兼容 Word/WPS 的 PAGE 复合域输出“第 N 页”，并在生成后的 DOCX/PDF 视觉检查中验证首、中、末页均显示页码。
- 建立代码材料选材与净化策略：只纳入可读、可追溯的业务源码；排除压缩样式和非业务构建文件；禁止 `AI UI Enhancer`、`LLM`、提示词、模型名等生成痕迹出现在任何提交材料中。文件边界使用中性的源码路径标识。
- 重做代码页排版：统一等宽字体、受控行宽和行距，长行按代码规则换行；按实际版心完成分页，避免空白大页，并在 DOCX 文本/XML 层和导出 PDF 层检查页密度。
- 取消用户手册中的“四格流程表”作为操作主体。改为每个菜单功能的“功能简介、适用对象/前提、操作说明（3–6 条完整步骤）、处理结果/注意事项、真实截图与图题”；登录与首页也按同一标准编写。
- 将上述要求加入自动校验：页码字段存在且可渲染、禁用 AI 标识零命中、代码页没有压缩 CSS 长行、每个功能单元有足量文字步骤及截图/图题；以临时项目生成 DOCX 后再导出 PDF 做视觉复核。

### 第二阶段验收标准

1. 设计说明书、用户操作手册和源代码材料均使用固定样式规范：标题层级、正文、代码、表格、图题、页眉页脚的中文/西文字体、字号、行距和对齐方式可被 DOCX 自动检查，并经 Word/WPS 打开后的视觉检查确认无默认字体回退、混乱字号或段落样式混用。
2. 打开 Word/WPS 后，每页页眉均显示完整的“第 N 页”，不存在只有“第”的页面。
3. 源代码材料全文不包含 `AI UI Enhancer`、`LLM`、模型名称、提示词等来源痕迹；不出现由单行压缩 CSS 引发的大面积留白。
4. 用户操作手册不再使用四格编号表替代操作说明；登录、首页和每个可见菜单模块都有不少于 3 条可执行的文字步骤、真实截图和图题。
5. 设计说明书、用户操作手册和源代码材料均通过生成后结构检查与 PDF 视觉检查；临时项目的前端构建和 JDK 17 Maven 测试仍通过。

# AI软著工厂开发 Harness

本规范用于用 Codex、Claude 或其他编码 Agent 持续开发 AI软著工厂。它约束的是**研发过程**，不嵌入用户侧的软著生成流程。

## 目标

- 让需求、代码、验证和交接可追踪。
- 将“分析、实施、复审”职责分开，避免 Agent 在未确认范围内自行扩展。
- 保护生成主链路：规划、项目生成、运行验证、Demo 审查、材料生成。
- 换窗口或换 Agent 后，仍能从仓库文件恢复完整上下文。

## 角色与边界

| 角色 | 默认负责人 | 必须产出 | 禁止行为 |
| --- | --- | --- | --- |
| 需求守门 | 用户 + Codex | ISSUE、范围、验收标准 | 未讨论直接改业务代码 |
| 架构与计划 | Codex | 影响面、实施步骤、风险、验证命令 | 以推测替代代码/日志复现 |
| 实施 | Claude 或指定编码 Agent | 限定范围的代码和测试 | 擅自扩大改动、重置用户改动 |
| 验证 | 实施者 + Codex | 单测、构建、必要的浏览器/生成器验证 | 只声明“已修复”而不运行验证 |
| 复审 | Codex | 按严重等级列出的风险与测试缺口 | 代替用户批准产品取舍 |
| 发布与交接 | Codex + 用户 | 文档同步、Git 提交范围、交接状态 | 混入未审查的无关改动 |

## 标准生命周期

```text
发现问题 / 新需求
  -> ISSUE 记录
  -> Codex 复现和方案
  -> 用户确认范围与验收
  -> 实施 Agent 修改
  -> 自动化验证
  -> Codex 复审
  -> 用户验收
  -> 文档同步与 Git 提交
```

### Gate 0：记录

用户指出问题时，先记录到 `docs/ISSUES.md`；如果该文件存在编码问题，新增单独的 `docs/ISSUE-XXX.md`，并在 `AGENTS.md` 标注入口。

每个 ISSUE 至少包含：

- 问题描述与复现条件。
- 影响范围和非目标。
- 初步根因或待确认项。
- 验收标准。
- 状态：`记录中`、`待实施`、`实施中`、`待复审`、`已修复`、`已关闭`。

**通过条件**：用户确认可以进入实施，或明确下达“实施 ISSUE-XXX / 统一修改”等命令。

### Gate 1：实施计划

Codex 在改动前给出简短、可执行的计划：

1. 需要修改的文件和职责。
2. 对状态机、生成产物、API 或 UI 的影响。
3. 回滚和兼容策略。
4. 验证命令及必要的人工验收点。

对高风险变更必须增加生成项目验证，不只验证工厂自身：

- `workflow.py`、`main.py`、`planner.py`：状态机/API/失败恢复测试。
- `project_generator.py`：生成项目构建、Maven 或合约检查。
- `frontend/src/pages/*`：前端构建和浏览器验证。
- 文档材料链路：至少验证产物存在、名称一致和合规报告。

### Gate 2：实施

实施 Agent 只能按确认计划修改。每个改动应具备：

- 最小化的代码变更。
- 对应测试，或写明为什么不适合自动化测试。
- 不覆盖、不回滚用户或其他 Agent 的未提交改动。
- 不直接修改 `outputs/` 中的历史任务来伪造通过结果。

实施完成时应报告：修改文件、行为变化、运行过的命令、未完成风险。

### Gate 3：验证

最低验证门禁：

```powershell
cd "C:\Users\whn\Documents\软著\backend"
python -m unittest discover -s tests -v

cd "C:\Users\whn\Documents\软著\frontend"
npm.cmd run build

cd "C:\Users\whn\Documents\软著"
git diff --check
```

按风险追加：

| 变更类型 | 额外验证 |
| --- | --- |
| Planner / Planning Review | 规划 JSON 校验、失败重试、人工审核页 |
| Project Generator | 临时生成项目、前端构建、JDK 17 Maven 测试 |
| Demo 运行 | 首页/历史任务启动、状态轮询、日志、停止与超时回收 |
| 材料和截图 | Demo 审查门禁、截图数量、文档/ZIP 和一致性报告 |
| UI | 浏览器检查关键页面、窄屏与常用交互 |

### Gate 4：Codex 复审

复审按以下顺序输出：

1. 阻断性问题：状态机跳步、数据丢失、生成项目不可运行、安全问题。
2. 行为回归：页面入口、路由、Demo 审查门禁、回退和恢复。
3. 测试缺口：哪些高风险路径尚未覆盖。
4. 简短改动摘要。

发现问题时，返回 Gate 1 或 Gate 2；无问题时标记 ISSUE 为 `待用户验收` 或 `已修复`。

### Gate 5：交接与版本化

每次完成一个功能批次必须同步：

- `README.md`：用户可见能力、配置、启动或使用方式变化。
- `docs/FLOW.md`：真实状态机、接口和执行顺序变化。
- `AGENTS.md`：当前结论、已完成升级、待办和验证结果。
- `docs/ISSUES.md` 或 `docs/ISSUE-XXX.md`：问题状态、根因、验收和验证结果。

提交前检查：

```powershell
git status --short
git diff --check
```

提交信息应围绕 ISSUE，例如：`Fix ISSUE-019 style enhancement timeout`。

## 当前工程的关键守卫

- `planning.json` 是代码、截图和文档的单一可信源。
- Demo 必须在截图和材料生成之前启动，并停在 `awaiting_demo_review` 等待用户确认。
- 返工必须先更新规划，再完整重新生成；禁止直接修补已生成项目作为正式方案。
- `outputs/` 是本地任务数据，除明确的恢复或验证操作外不得删除或伪造。
- 生成器、Demo 和材料链路的变化必须考虑 JDK 17、前端构建和任务恢复。

## 任务交接模板

新 Agent 接手或将任务交给 Claude 时，使用下面模板：

```text
任务：ISSUE-XXX <标题>
状态：待实施 / 待复审
背景：<用户反馈和复现条件>
范围：<允许修改的文件/模块>
非目标：<明确不做的事>
验收：<可验证结果>
验证命令：<单测、构建、浏览器或生成项目检查>
注意：保留当前工作区改动；不要修改 outputs 历史任务；完成后报告 diff 和测试结果。
```

## 快速分工示例

```text
Codex：登记 ISSUE -> 复现 -> 冻结计划 -> 复审 -> 更新交接文档
Claude：依据冻结计划实施 -> 补测试 -> 报告修改和验证
用户：确认产品取舍、授权实施、验收最终体验
```

这套 Harness 的原则是：**先把变更变成可审查的事实，再把事实变成代码。**

## Claude 实施交接

不在当前工厂进程中包装或托管 Claude。需要实施时，由用户在新的 Claude Code 窗口直接执行；Harness 通过仓库上下文保证接手一致性。

新窗口按以下顺序读取：

1. `AGENTS.md`
2. `docs/HARNESS.md`
3. 对应 `docs/ISSUE-XXX.md`
4. 受影响的代码、测试和流程文档

向 Claude 交接时使用本文件的“任务交接模板”，并明确：允许修改路径、非目标、验收命令、不得提交/推送/回滚其他改动。Claude 完成后由 Codex 读取 Git diff、运行验证并复审。

这种方式避免了当前会话超时中断本地 Claude 进程，也不需要额外的后台进程、轮询或自动通知机制。
